# 運用上のガードレール

このリポジトリは Claude Code の実行時制御に集中します。そのうえで、設定やフックを補助するプロジェクト指示の例も置きます。

## `.claudeignore` の例を置かない理由

このリポジトリでは、機密ファイル保護を Claude Code 設定側で扱います。

- `permissions.deny`
- サンドボックスの `filesystem.denyRead`
- 一般的な機密パスを検出する Bash PreToolUse フック

これにより、有効なポリシーを `.claude/settings.json` に集約できます。

既に組織で `.claudeignore` を使っている場合は併用して構いません。ただし、このリポジトリの基準は `.claudeignore` に依存しません。ここでの基準になる設定は `.claude/settings.json` です。

## CLAUDE.md

`claude/CLAUDE.example.md` はプロジェクトメモリの例です。これは強制の仕組みではありません。Claude Code に対して、プロジェクト固有の行動指針を与えます。

- 機密値を出力しない
- 認証情報をソースコードに埋め込まない
- 個人データ / 顧客データをログに出さない
- 本番に近い操作では明示的な承認を要求する

対象アプリの領域に合わせて編集して使います。

## 本番環境の値

本番環境の値は、可能な限りアプリの作業領域の外で管理します。

推奨配置:

```text
your-app/
  .env.example          # ダミー値。Claude Code が読んでよい
  .env                  # ローカル開発用。この基準では拒否
  secrets/              # この基準では拒否
    .env.production     # 本番に近い値。Claude Code の対象外
```

デプロイには CI/CD 環境変数やホスティングプロバイダのシークレットストアを使います。Claude Code には生の本番値ではなく、例、ローカルのダミー値、設定名のドキュメントを見せます。

## 未知のリポジトリを信頼する前の確認

自分が作ったものではないリポジトリを信頼する前に、Claude Code 関連ファイルを確認します。この基準はそれらをレビューしやすくするためのものですが、リポジトリ側に独自のローカルポリシーが入っていることがあります。

確認するもの:

- `.claude/settings.json` に広すぎる `allow`、サンドボックス無効化、バイパス設定の緩和がないか
- `.claude/hooks/` にツール利用の前後で実行されるスクリプトがないか
- `.claude/skills/` に外部送信、本番アクセス、認証情報の取り扱いを促す指示がないか
- MCP サーバー定義が未知のエンドポイントやコマンドを向いていないか
- プロジェクト `CLAUDE.md` がチームポリシーと矛盾していないか

意外な `.claude/` 挙動があるリポジトリは、`/status` と `/permissions` で有効な設定を理解するまで信頼しません。

## MCP、プラグイン、スキル

MCP サーバー、プラグイン、スキルは、Claude Code ができることや振る舞いを拡張します。このリポジトリは MCP の信頼経路制御を実装しません。実行時の権限設定とフックを中心に、Claude Code のローカル実行制御の例を提供します。

共有環境では次を推奨します。

- 既知または公式の MCP サーバー、プラグイン、スキルを優先する
- MCP サーバー設定は小さくレビューしやすく保つ
- 組織として許可対象を制御したい場合は `allowManagedMcpServersOnly` などの管理設定を使う
- スキルは指示であり強制ではないものとして扱う

## スキル

`claude/skills/db-change-review/skill.md` は高リスクなデータベース作業向けの軽量な承認フロー例です。

スキルは振る舞いの指示です。権限設定、サンドボックス、フック、人間のレビューの代替ではありません。「ツール呼び出しを無条件に拒否したい」場合ではなく、「一度止まり、影響範囲を整理し、承認を求めてほしい」場合に使います。
